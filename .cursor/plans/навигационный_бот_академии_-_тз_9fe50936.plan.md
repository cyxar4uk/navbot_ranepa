# Техническое задание: Навигационная платформа Академии

## 1. Общее описание продукта

**Навигационная платформа Академии** — это Telegram Bot + Telegram Mini App для сопровождения мероприятий. Платформа позволяет посетителям навигироваться по Академии, просматривать программу мероприятий, записываться на события, получать помощь от AI-ассистента. Администраторы могут полностью настраивать контент и структуру приложения через админ-панель без участия разработчиков.

### Ключевые особенности

- Модульная архитектура: все функции представлены как настраиваемые модули
- Контекст события: приложение работает в рамках конкретного мероприятия
- Полная настройка через админку: включение/выключение модулей, порядок, конфигурация
- Два профиля: пользователь и администратор
- AI-ассистент с контекстом события
- Интерактивная карта Академии
- Интеграция внешних ссылок

## 2. Продуктовая модель

### 2.1. Центральная сущность — Event

Вся работа приложения происходит в контексте события (мероприятия).

**Event** содержит:

- `id` — уникальный идентификатор
- `title` — название мероприятия
- `description` — описание
- `date_start` — дата начала
- `date_end` — дата окончания
- `location` — основное место проведения
- `status` — статус (upcoming / active / finished)
- `created_at`, `updated_at` — временные метки

### 2.2. Модульная система

Каждое событие состоит из набора модулей. Модули определяют функциональность и отображаются на главном экране (Dashboard) как кнопки.

**Module** содержит:

- `id` — уникальный идентификатор
- `event_id` — привязка к событию
- `type` — тип модуля (см. раздел 2.3)
- `title` — название для отображения
- `icon` — иконка (emoji или код)
- `enabled` — включен/выключен
- `order` — порядок отображения
- `badge_type` — тип бейджа (none / count / dot)
- `badge_value` — значение бейджа (опционально)
- `config` — JSON-конфигурация модуля
- `created_at`, `updated_at` — временные метки

### 2.3. Типы модулей

#### 2.3.1. ProgramModule

Программа мероприятия (деловая, культурная, по дням).

**Config:**

```json
{
  "grouping": "day" | "track",
  "allow_registration": boolean,
  "show_capacity": boolean,
  "filters_enabled": boolean,
  "event_types": ["string"],
  "default_view": "timeline" | "list"
}
```

#### 2.3.2. EventListModule

Универсальный список сущностей (спикеры, участники, партнеры, стартапы).

**Config:**

```json
{
  "entity_type": "speaker" | "participant" | "partner" | "startup",
  "card_layout": "compact" | "full",
  "searchable": boolean,
  "filterable": boolean,
  "show_details": boolean
}
```

#### 2.3.3. MapModule

Интерактивная карта Академии с навигацией.

**Config:**

```json
{
  "map_type": "svg" | "image",
  "default_floor": number,
  "zones_enabled": boolean,
  "link_with_events": boolean,
  "show_navigation": boolean
}
```

#### 2.3.4. RegistrationModule

Запись на мероприятия/воркшопы.

**Config:**

```json
{
  "max_capacity": number,
  "approval_required": boolean,
  "show_remaining": boolean,
  "waitlist_enabled": boolean
}
```

#### 2.3.5. ExternalLinkModule

Подгрузка сторонних ресурсов.

**Config:**

```json
{
  "url": "string",
  "open_type": "webview" | "external",
  "auth_required": boolean,
  "title": "string"
}
```

#### 2.3.6. CustomPageModule

Статические страницы (FAQ, правила, инструкции).

**Config:**

```json
{
  "content_type": "markdown" | "html",
  "content": "string",
  "editable": boolean
}
```

#### 2.3.7. AssistantModule

Контекстный AI-ассистент.

**Config:**

```json
{
  "scope": "event" | "module" | "item",
  "allow_free_questions": boolean,
  "fallback_to_operator": boolean,
  "system_prompt_template": "string"

```

#### 2.3.8. NewsModule

Новости и анонсы мероприятия.

**Config:**

Личные сообщения пользователя.

**Config:**

```json
}
```

## 3. Модель данных

### 3.1. Основные таблицы

#### events

```sql
CREATE TABLE events (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title VARCHAR(255) NOT NULL,
    description TEXT,
    date_start TIMESTAMP NOT NULL,
    date_end TIMESTAMP NOT NULL,
    location VARCHAR(255),
    status VARCHAR(20) DEFAULT 'upcoming',
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
```

#### modules

```sql
CREATE TABLE modules (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    event_id UUID REFERENCES events(id) ON DELETE CASCADE,
    type VARCHAR(50) NOT NULL,
    title VARCHAR(255) NOT NULL,
    icon VARCHAR(10),
    enabled BOOLEAN DEFAULT true,
    order INTEGER DEFAULT 0,
    badge_type VARCHAR(20) DEFAULT 'none',
    badge_value VARCHAR(50),
    config JSONB DEFAULT '{}',
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
```

#### event_items

```sql
CREATE TABLE event_items (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    event_id UUID REFERENCES events(id) ON DELETE CASCADE,
    module_id UUID REFERENCES modules(id) ON DELETE SET NULL,
    title VARCHAR(255) NOT NULL,
    description TEXT,
    date_start TIMESTAMP,
    date_end TIMESTAMP,
    location_id UUID REFERENCES locations(id),
    capacity INTEGER,
    registered_count INTEGER DEFAULT 0,
    type VARCHAR(50),
    status VARCHAR(20) DEFAULT 'active',
    metadata JSONB DEFAULT '{}',
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
```

#### speakers

```sql
CREATE TABLE speakers (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    event_id UUID REFERENCES events(id) ON DELETE CASCADE,
    name VARCHAR(255) NOT NULL,
    bio TEXT,
    photo_url VARCHAR(500),
    position VARCHAR(255),
    company VARCHAR(255),
    social_links JSONB DEFAULT '{}',
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
```

#### event_speakers (связь многие-ко-многим)

```sql
CREATE TABLE event_speakers (
    event_item_id UUID REFERENCES event_items(id) ON DELETE CASCADE,
    speaker_id UUID REFERENCES speakers(id) ON DELETE CASCADE,
    PRIMARY KEY (event_item_id, speaker_id)
);
```

#### registrations

```sql
CREATE TABLE registrations (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    event_item_id UUID REFERENCES event_items(id) ON DELETE CASCADE,
    user_id UUID REFERENCES users(id) ON DELETE CASCADE,
    status VARCHAR(20) DEFAULT 'pending',
    registered_at TIMESTAMP DEFAULT NOW(),
    approved_at TIMESTAMP,
    UNIQUE(event_item_id, user_id)
);
```

#### users

```sql
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    telegram_id BIGINT UNIQUE NOT NULL,
    username VARCHAR(255),
    first_name VARCHAR(255),
    last_name VARCHAR(255),
    role VARCHAR(20) DEFAULT 'user',
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
```

#### locations

```sql
CREATE TABLE locations (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    event_id UUID REFERENCES events(id) ON DELETE CASCADE,
    name VARCHAR(255) NOT NULL,
    description TEXT,
    floor INTEGER,
    zone_id UUID REFERENCES zones(id),
    coordinates JSONB,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
```

#### zones

```sql
CREATE TABLE zones (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    event_id UUID REFERENCES events(id) ON DELETE CASCADE,
    name VARCHAR(255) NOT NULL,
    floor INTEGER,
    coordinates JSONB,
    map_data JSONB,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
```

#### assistant_knowledge

```sql
CREATE TABLE assistant_knowledge (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    event_id UUID REFERENCES events(id) ON DELETE CASCADE,
    content_type VARCHAR(50),
    content TEXT NOT NULL,
    metadata JSONB DEFAULT '{}',
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
```

#### news

```sql
CREATE TABLE news (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    event_id UUID REFERENCES events(id) ON DELETE CASCADE,
    title VARCHAR(255) NOT NULL,
    content TEXT,
    image_url VARCHAR(500),
    published_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
```

#### messages

```sql
CREATE TABLE messages (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    event_id UUID REFERENCES events(id) ON DELETE CASCADE,
    from_user_id UUID REFERENCES users(id) ON DELETE CASCADE,
    to_user_id UUID REFERENCES users(id) ON DELETE SET NULL,
    content TEXT NOT NULL,
    read_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW()
);
```

## 5. API спецификация

### 5.1. Аутентификация

Все запросы требуют заголовок:

```
Authorization: Bearer <telegram_init_data>
```

Backend валидирует `telegram_init_data` через Telegram WebApp API.

### 5.2. Endpoints

#### Events

**GET /api/events**

Получить список событий.

**GET /api/events/{event_id}**

Получить детали события.

**GET /api/events/active**

Получить активное событие (по дате).

#### Modules

**GET /api/events/{event_id}/modules**

Получить модули события (только enabled, отсортированные по order).

**GET /api/modules/{module_id}**

Получить детали модуля.

**PUT /api/modules/{module_id}**

Обновить модуль (только для админов).

**POST /api/modules**

Создать модуль (только для админов).

**DELETE /api/modules/{module_id}**

Удалить модуль (только для админов).

**PUT /api/modules/reorder**

Обновить порядок модулей (только для админов).

#### Event Items (программа)

**GET /api/events/{event_id}/items**

Получить события программы.

Query params:

- `day` — фильтр по дню
- `type` — фильтр по типу
- `location` — фильтр по месту
- `search` — поиск по названию
- `available_only` — только с доступными местами

**GET /api/event-items/{item_id}**

Получить детали события программы.

**POST /api/event-items/{item_id}/register**

Записаться на событие.

**DELETE /api/event-items/{item_id}/register**

Отменить запись.

#### Speakers

**GET /api/events/{event_id}/speakers**

Получить спикеров события.

**GET /api/speakers/{speaker_id}**

Получить детали спикера.

#### Map

**GET /api/events/{event_id}/map**

Получить данные карты.

**GET /api/locations/{location_id}**

Получить детали локации.

#### Assistant

**POST /api/assistant/chat**

Отправить сообщение ассистенту.

Body:

```json
{
  "event_id": "uuid",
  "message": "string",
  "context": {
    "module_id": "uuid",
    "item_id": "uuid"
  }
}
```

Response:

```json
{
  "response": "string",
  "sources": ["string"]
}
```

#### Admin

**GET /api/admin/events**

Получить все события (только для админов).

**POST /api/admin/events**

Создать событие.

**PUT /api/admin/events/{event_id}**

Обновить событие.

**DELETE /api/admin/events/{event_id}**

Удалить событие.

**GET /api/admin/modules/types**

Получить доступные типы модулей.

**POST /api/admin/content**

Создать/обновить контент (спикеры, новости, локации).

## 6. UX-флоу

### 6.1. Пользовательский флоу

#### Entry Flow

1. Пользователь открывает бота в Telegram
2. Бот отправляет кнопку "Открыть приложение"
3. Пользователь открывает Mini App
4. Приложение определяет активное событие (по дате или выбор)
5. Загружается Dashboard с модулями события

#### Dashboard Flow

1. Отображается сетка модулей (только enabled, по order)
2. Пользователь кликает на модуль
3. Открывается соответствующий экран модуля

#### Program Flow

1. Пользователь открывает ProgramModule
2. Отображается список дней или треков (в зависимости от config)
3. Пользователь выбирает день
4. Отображается Timeline с событиями дня
5. Пользователь может:

   - Применить фильтры (тип, место, доступность)
   - Использовать поиск
   - Открыть карточку события

6. На карточке события:

   - Просмотр деталей
   - Запись (если доступна)
   - Переход на карту (если есть место)
   - Задать вопрос ассистенту

#### Map Flow

1. Пользователь открывает MapModule
2. Отображается карта Академии
3. Пользователь может:

   - Выбрать этаж
   - Кликнуть на зону/локацию
   - Получить информацию о локации
   - Перейти к событиям в этой локации

#### Assistant Flow

1. Пользователь открывает AssistantModule
2. Отображается чат-интерфейс
3. Ассистент имеет контекст:

   - Текущее событие
   - Текущий модуль (если открыт из модуля)
   - Текущий item (если открыт из карточки события)

4. Пользователь задает вопрос
5. Ассистент отвечает на основе данных БД
6. При необходимости — fallback на оператора

### 6.2. Администраторский флоу

#### Admin Entry

1. Администратор открывает бота
2. Проверка роли (telegram_id в таблице users с role='admin')
3. Доступ к админ-панели

#### Event Builder Flow

1. Администратор выбирает событие или создает новое
2. Открывается Event Builder:

   - Список всех доступных типов модулей
   - Список включенных модулей события

3. Администратор может:

   - Включить/выключить модуль (toggle)
   - Изменить порядок (drag & drop)
   - Настроить конфигурацию модуля
   - Создать новый модуль

4. Изменения сохраняются и применяются сразу

#### Content Management Flow

1. Администратор переходит в раздел Content
2. Может управлять:

   - Событиями программы (event_items)
   - Спикерами
   - Новостями
   - Локациями и зонами
   - Знаниями для ассистента

## 10. Риски и митигация

### Технические риски

- **LLM API недоступен**: Fallback на оператора или предопределенные ответы
- **Производительность при большом количестве событий**: Кэширование, пагинация
- **Ограничения Telegram Mini App**: Соблюдение лимитов размера, оптимизация

### Продуктовые риски

- **Сложность настройки для админов**: Подробная документация, видео-инструкции
- **Недостаточная гибкость модулей**: Расширяемая JSON-конфигурация
